# CLAUDE.md — Игра «Инспектор входящих»

> **Назначение файла.** Это единственный источник истины и техническое задание для реализации игры. Построй по нему полностью рабочую игру, **не отступая от сценария и не пропуская ни одной детали**. Все тексты писем, тем, отправителей, вложений и диалогов бери **дословно** из раздела с данными — это игра про информационную безопасность, и любая опечатка вроде `techn0sphera`, `yandx`, `.pdf.exe` является намеренным обучающим элементом, её нельзя «исправлять».

---

## 1. Задача

Реализовать текстовый симулятор-свайпер «Инспектор входящих»: игрок в роли стажёра отдела информационной безопасности (ИБ) белорусской ИТ-компании «ТехноСфера» разбирает входящую почту в течение рабочей недели (5 дней), отправляя фишинг в карантин и пропуская легитимные письма. Игра должна быть **полностью проходимой от начала (заставка) до одной из концовок**.

Игра одностраничная, работает в браузере, поддерживает и **свайпы** (мобильные/тач и мышь), и **кнопки** (десктоп). Никакого бэкенда, никаких внешних ассетов (картинок/шрифтов из сети) — всё самодостаточно.

---

## 2. Технологический стек и запуск

- **Рекомендуемый стек:** React + Vite + TypeScript. Стили — обычный CSS (CSS-модули или один глобальный файл) либо Tailwind — на твоё усмотрение, но интерфейс должен выглядеть как корпоративный почтовый клиент (см. раздел 14).
- **Анимация свайпа:** `framer-motion` (drag + вылет карточки за экран) либо собственная реализация на Pointer Events. Свайп и кнопки должны приводить к одному и тому же результату.
- **Хранение прогресса:** опционально `localStorage` для сохранения лучшего результата/последней концовки (в проекте Claude Code это допустимо; браузерное хранилище работает нормально).
- Проект должен запускаться командами `npm install` и `npm run dev`, и собираться `npm run build`. Добавь короткий `README.md` с инструкцией запуска.
- Если по каким-то причинам React неудобен — допустима реализация на чистом HTML/CSS/JS (один `index.html` без сборки). Функциональность и тексты при этом должны быть идентичны.

## 4. Концепция и сеттинг

- **Жанр:** текстовый симулятор, карточная головоломка.
- **Сеттинг:** офис крупной белорусской ИТ-компании **«ТехноСфера»**. Интерфейс стилизован под внутренний корпоративный почтовый клиент.
- **Главный герой (игрок):** младший специалист по информационной безопасности (стажёр).
- **Цель игрока:** продержаться рабочую неделю (5 дней), отсеивая фишинговые письма и пропуская легитимные рабочие сообщения.

---

## 5. Действующие лица (NPC)

- **Анна Сергеевна** — начальница ИБ. Строгая, опытная. Появляется **в начале и в конце каждого дня**: даёт инструкции, хвалит за успехи и отчитывает за критические ошибки. Речь строгая, деловая, требовательная.
- **Виталий** — системный администратор. Добродушный коллега. Время от времени присылает подсказки и предупреждает о новых типах кибератак. Речь дружелюбная, разговорная. В этой версии Виталий ведёт вводный инструктаж в **среду**.

Обращение к игроку — «стажёр». Пол игрока не задан: в репликах-«шаблонах» (конец дня) используй нейтральные формулировки, без родовых окончаний глаголов.

## 6. Игровые механики и интерфейс

**Карточка письма.** На экране — карточка входящего письма. Она показывает: адрес отправителя (крупно, т.к. анализ домена — часть геймплея), тему, текст письма и, если есть, вложение с именем файла и его расширением.

**Действия игрока (два способа, эквивалентны):**
- **Свайп вправо** или кнопка **«Пропустить»** → письмо доставляется адресату. Внутреннее действие: `pass`.
- **Свайп влево** или кнопка **«В карантин»** → письмо блокируется как подозрительное. Внутреннее действие: `quarantine`.

Свайп сопровождается анимацией: карточка следует за пальцем/курсором, наклоняется, при достижении порога подсвечивается зелёным (вправо/пропустить) или красным (влево/карантин) и вылетает за экран. Кнопки запускают ту же анимацию в соответствующую сторону.

**Показатели на экране (HUD), всегда видны:**
- Шкала **«Безопасность»** (0–100 %). Падает, если игрок **пропускает фишинг** в сеть компании.
- Шкала **«Продуктивность»** (0–100 %). Падает, если игрок **блокирует важное рабочее письмо** (отправляет в карантин легитимное).
- **Индикатор времени/прогресса:** текущий день недели + количество писем, например «Понедельник, писем: 3/10». Формат — `текущее/всего`, где «всего» = число карточек, показываемых в этом дне.

**Обратная связь после решения.** После каждого свайпа коротко покажи реакцию: верно/ошибка, как изменились шкалы, и (для фишинга) пояснение, почему это угроза. Пояснения бери из данных (поле `explanation`) там, где они заданы. Для карточек без пояснения можешь **опционально** добавить короткую обучающую подсказку в духе темы дня (см. раздел 15) — это допустимое улучшение, но исходные тексты писем менять нельзя.

## 7. Экраны и поток игры (game flow)

1. **Заставка (StartScreen):** название игры «Инспектор входящих», компания «ТехноСфера», короткое описание роли стажёра ИБ, кнопка «Начать смену». Упоминание команды-автора: SPDubstep (в подвале/титрах).
2. **Вводный экран дня (DayIntro):** имя и реплика NPC (Анна — пн, вт, чт, пт; Виталий — ср) + заголовок и цель дня. Кнопка «К письмам».
3. **Геймплей дня:** колода карточек этого дня в перемешанном порядке (безопасные и фишинговые вперемешку). Игрок разбирает письма свайпами/кнопками. HUD обновляется в реальном времени. Если любая шкала достигает 0 % — немедленно переход к соответствующей концовке-поражению (см. раздел 8), день не доигрывается.
4. **Итог дня (DaySummary):** сколько писем разобрано, сколько ошибок, текущие значения шкал + реплика Анны Сергеевны (похвала/предупреждение/выговор по числу ошибок за день). Кнопка «Следующий день» (или «Домой на выходные» после пятницы).
5. **Концовка (EndingScreen):** один из финалов из раздела 12, с кнопкой «Играть заново» (сброс состояния).

Переходы между экранами — плавные. После пятницы (если ни одна шкала не обнулилась) — проверка условий и показ итоговой концовки.

## 8. Логика очков и условия победы/поражения

**Стартовые значения.** Обе шкалы стартуют со **100 %**.

**Персистентность.** Шкалы **сквозные на всю неделю** — переносятся из дня в день и **не сбрасываются** между днями. Итоговая победа проверяется в конце пятницы, поражение может наступить в любой из дней.

**Изменение шкал за одно решение (значения по умолчанию, см. пометку о настройке ниже):**

| Тип письма | Действие игрока | Верно? | Эффект |
|---|---|---|---|
| Безопасное (`safe`) | `pass` (пропустить) | ✔ верно | без штрафа |
| Безопасное (`safe`) | `quarantine` (в карантин) | ✗ ошибка | **Продуктивность −10** |
| Фишинг (`phishing`) | `quarantine` (в карантин) | ✔ верно | без штрафа |
| Фишинг (`phishing`) | `pass` (пропустить) | ✗ ошибка | **Безопасность −10** |

Значения зажимаются в диапазон 0…100.

> **Пометка о настройке (важно для баланса).** Величина штрафа `−10` — это дефолт, вынеси её в константы (`PENALTY = 10`). При `−10` победа (обе шкалы >70 %) требует не более 2 ошибок каждого типа за всю неделю, а поражение (0 %) наступает при ~10 ошибках одного типа. Если нужен другой баланс — меняй только константу. Опционально можно добавить микро-восстановление за верное решение (`+1`, но не выше 100) — оставляю на усмотрение, по умолчанию верное решение шкалу не меняет.

**Условия окончания:**

- **Поражение по безопасности:** как только «Безопасность» достигает **0 %** в любой день → концовка 2 «Взлом системы» (немедленно).
- **Поражение по продуктивности:** как только «Продуктивность» достигает **0 %** в любой день → концовка 3 «Бюрократический тупик» (немедленно).
- **Победа:** дошли до конца пятницы и **обе** шкалы **строго выше 70 %** → концовка 1 «Успешный специалист».
- **Граничный случай (нейтральный финал):** дошли до конца пятницы, ни одна шкала не обнулилась, но хотя бы одна ≤ 70 % → см. **концовку 4** в разделе 12. *Это добавленный финал, чтобы автомат состояний был полным; трёх канонических концовок из сценария он не отменяет и включается только в этой «серой зоне».*

**Приоритет проверок:** сначала проверяй обнуление шкал (мгновенные поражения) на каждом решении; проверку победы/нейтрального финала делай один раз — после последнего письма пятницы.

**Число карточек в дне.** По умолчанию день показывает **всю свою колоду** в перемешанном порядке: понедельник–четверг по 14 писем, пятница — 24 письма (пятница — «завал», намеренно длиннее и с более плотным потоком). Счётчик «X/Y» использует это Y. При желании можно ограничить день выборкой из ~10–12 писем (как в примере «3/10») со случайным набором из пула дня — вынеси это в настройку, но по умолчанию используй полные колоды, чтобы не терять контент. Пятница обязательно должна ощущаться быстрее/плотнее прочих дней (больше писем, смешанные типы угроз); опционально можно добавить лёгкий таймер на карточку именно в пятницу.

## 9. Дни: заголовки, цели, вводные диалоги, конец дня

Заведи это в `days.ts`. Вводные реплики приведены **дословно**; реплики конца дня — рекомендованные (сценарий требует появления Анны в конце дня, но точных слов не даёт — можешь оставить как есть или слегка доработать в её строгом стиле, смысл сохраняя).

### День 1 — Понедельник · «Курс молодого бойца» / «Базовый уровень»
- **Цель дня:** понять базовые отличия писем. Обучение.
- **Вводная (Анна Сергеевна):** «Добро пожаловать в команду, стажёр. Твоя задача – защитить почту сотрудников от внешних угроз. Но помни: если ты заблокируешь важные контракты клиентов, бизнес встанет. Начнём с простого».

### День 2 — Вторник · «Дьявол в деталях»
- **Цель дня:** научить игрока внимательно смотреть на доменные имена отправителей (тайпосквоттинг).
- **Вводная (Анна Сергеевна):** «Вчера было просто. Обращай внимание на адреса отправителей. Буква "о" и цифра "0" – разные вещи».

### День 3 — Среда · «Эмоциональные качели»
- **Цель дня:** научить распознавать психологическое давление (срочность, угрозы, страх).
- **Вводная (Виталий, сисадмин):** «Привет! Если в письме кричат "СРОЧНО!", "ШТРАФ!" или "АККАУНТ ЗАБЛОКИРОВАН В ТЕЧЕНИЕ 2 ЧАСОВ" — это почти всегда развод. Не поддавайся панике».

### День 4 — Четверг · «Опасные файлы»
- **Цель дня:** фокус на анализе вложений (двойные расширения, исполняемые файлы).
- **Вводная (Анна Сергеевна):** «Мошенники часто прячут вирусы во вложениях. Помни: под видом документа .pdf или .docx может скрываться вредоносная программа с двойным расширением».

### День 5 — Пятница · «Пятничный завал»
- **Цель дня:** проверка всех полученных навыков. Письма идут быстрее, типы угроз смешаны.
- **Вводная (Анна Сергеевна):** «Конец недели, все расслабились и спешат домой. Это идеальное время для хакеров. Соберись, от этого зависит, пойдёшь ли ты на выходные или проведёшь их, восстанавливая серверы компании».

### Реплики Анны в конце дня (рекомендованные, по числу ошибок за день)
- **0 ошибок:** «Смена отработана чисто, стажёр. Ни одной ошибки. Но расслабляться рано».
- **1–2 ошибки:** «Пара писем ушла не туда. В нашем деле это уже риск. Завтра — внимательнее».
- **3 и более:** «Сегодня много ошибок, стажёр. Каждое такое письмо — брешь в обороне компании. Ещё один такой день — и разговор будет другим».

## 10. Модель данных (TypeScript)

```ts
// src/game/types.ts
export type CardType = 'safe' | 'phishing';
export type Action = 'pass' | 'quarantine'; // pass = свайп вправо; quarantine = свайп влево

export interface Card {
  id: string;            // напр. "1.1"
  day: 1 | 2 | 3 | 4 | 5;
  type: CardType;
  sender: string;        // e-mail отправителя (показывать дословно, домен важен)
  subject: string;       // тема письма
  body: string;          // текст письма
  attachment: string | null; // имя файла с расширением, либо null
  correct: Action;       // safe → 'pass', phishing → 'quarantine'
  explanation: string | null; // «Пояснение» из сценария, где задано
}

export interface DayConfig {
  day: 1 | 2 | 3 | 4 | 5;
  weekday: string;       // "Понедельник" ...
  title: string;         // "Базовый уровень" ...
  goal: string;
  introSpeaker: 'Анна Сергеевна' | 'Виталий';
  introText: string;
}

export type Outcome =
  | 'win'                 // концовка 1
  | 'lose_security'       // концовка 2
  | 'lose_productivity'   // концовка 3
  | 'neutral';            // концовка 4 (граничный случай)

export interface GameState {
  security: number;       // 0..100, старт 100
  productivity: number;   // 0..100, старт 100
  currentDay: 1 | 2 | 3 | 4 | 5;
  queue: Card[];          // перемешанная колода текущего дня
  index: number;          // индекс текущей карточки
  mistakesToday: number;
  finished: boolean;
  outcome: Outcome | null;
}
```

Инвариант данных: для `type: 'safe'` всегда `correct: 'pass'`; для `type: 'phishing'` всегда `correct: 'quarantine'`. Ошибкой считается несовпадение действия игрока с `correct`.

## 11. Концовки (полные тексты)

Занеси в `endings.ts`.

### Концовка 1 — «Успешный специалист» (Победа)
- **Условие:** обе шкалы («Безопасность» и «Продуктивность») строго выше 70 % на конец пятницы.
- **Финал:** Анна Сергеевна жмёт вам руку. «Отличная работа, стажёр. Ты уберёг компанию от трёх серьёзных атак и не сорвал ни одной сделки. Поздравляю, ты принят в штат на должность младшего аналитика ИБ!»
- Визуал: спокойный, «успешный» тон.

### Концовка 2 — «Взлом системы» (Поражение по безопасности)
- **Условие:** шкала «Безопасность» упала до 0 % в любой из дней.
- **Финал:** экран становится красным. Появляется баннер программы-вымогателя: «Все ваши файлы зашифрованы». Затем Анна Сергеевна: «Из-за твоей халатности вирус-шифровальщик парализовал работу всей компании. Убытки исчисляются миллионами. Ты уволен, и, кажется, нам придётся встретиться в суде».
- Визуал: полностью красный экран, стилизованный баннер ransomware.

### Концовка 3 — «Бюрократический тупик» (Поражение по продуктивности)
- **Условие:** шкала «Продуктивность» упала до 0 % в любой из дней.
- **Финал:** на экране появляется разгневанный директор компании. Анна Сергеевна: «Ты параноик, стажёр. Ты отправил в карантин контракты с ключевыми инвесторами, счета на оплату электричества и даже поздравление директора с днём рождения. Бизнес парализован, потому что никто не получает почту. ИБ должно помогать работе, а не блокировать её. Ты нам не подходишь».

### Концовка 4 — «Испытательный срок продолжается» (нейтральный, граничный случай — добавленный)
- **Условие:** дошли до конца пятницы, ни одна шкала не обнулилась, но хотя бы одна ≤ 70 %.

- Пометка: этот финал не входит в три канонические концовки сценария и служит только для полноты (см. раздел 8).

## 13. Дизайн и визуальный стиль

Интерфейс — **внутренний корпоративный почтовый клиент «ТехноСфера»**. Ориентиры:

- Общий вид: аккуратный деловой веб-клиент почты. Шапка с логотипом/названием «ТехноСфера · Почта» и подписью роли «Отдел ИБ · Стажёр». Никаких сторонних изображений — логотип можно собрать из текста/CSS/inline-SVG.
- **Карточка письма:** блок с чётко выделенным адресом отправителя (моноширинный шрифт для адреса, чтобы были видны подмены в домене), строкой темы, телом письма и, если есть, «плашкой вложения» — иконка файла + имя файла с расширением. Для опасных вложений расширение всё равно показывается как есть (`.pdf.bat`, `.xlsx.exe`, `.js`, `.docm`, `.xlsm`, `.exe`), игрок должен сам заметить.
- **Свайп/кнопки:** внизу две крупные кнопки — «В карантин» (слева, красная) и «Пропустить» (справа, зелёная). При свайпе — подсветка направления и метка («ПРОПУСТИТЬ» / «В КАРАНТИН») на карточке.
- **HUD:** две горизонтальные шкалы-прогрессбары «Безопасность» и «Продуктивность» с процентами и цветовой индикацией (зелёный → жёлтый → красный по мере падения). Рядом — «Понедельник · писем 3/10».
- **Реакция после решения:** короткий тост/панель: «Верно» (зелёный) или «Ошибка» (красный) + что произошло (например, «Пропущен фишинг: Безопасность −10») + пояснение.
- **Концовки:** отдельные полноэкранные состояния. Концовка 2 — красный экран с баннером ransomware «Все ваши файлы зашифрованы». Концовка 1 — спокойный/светлый успешный экран. Стиль должен усиливать эмоцию финала.
- Адаптивность: корректно работает и на телефоне (свайп), и на десктопе (кнопки + свайп мышью). Используй системные шрифты, чтобы не тянуть ничего из сети.

Если используешь `frontend-design` подход — придерживайся сдержанной корпоративной палитры (сине-серый как основной, зелёный/красный как акценты для действий и шкал). Аккуратно, читаемо, без визуального шума.

## 17. Definition of Done (чек-лист готовности)

Игра считается готовой, когда:

1. Проект запускается (`npm install` + `npm run dev`) и собирается (`npm run build`); есть краткий README.
2. Реализованы все экраны: заставка → вводная дня → геймплей → итог дня → концовка; с кнопкой «Играть заново».
3. Присутствуют **все 5 дней** с корректными заголовками, целями и **дословными** вводными репликами (Анна — пн/вт/чт/пт; Виталий — ср).
4. В `cards.ts` **все 80 карточек** из раздела 12, дословно, с верными `type`/`correct` и вложениями. Дни 1–4 по 14 карточек, пятница — 24.
5. Работают **оба** способа решения — свайп (тач + мышь) и кнопки «Пропустить»/«В карантин» — с анимацией и подсветкой направления.
6. HUD показывает обе шкалы «Безопасность» и «Продуктивность» (0–100 %), день недели и счётчик «X/Y»; обновляется мгновенно.
7. Корректная логика очков: старт 100/100, сквозные шкалы на всю неделю, штраф −10 (в константе) за ошибку нужного типа, зажим 0…100.
8. Условия финалов: обнуление «Безопасности» → концовка 2; обнуление «Продуктивности» → концовка 3; обе >70 % в конце пятницы → концовка 1; иначе (граничный случай) → концовка 4. Тексты концовок 1–3 — дословно из раздела 11.
9. Анна появляется **в конце каждого дня** с репликой по числу ошибок; после решения показывается реакция (верно/ошибка + пояснение).
10. Внешний вид — корпоративный почтовый клиент «ТехноСфера»; адрес отправителя читаемо выделен (видны подмены домена); опасные расширения вложений отображаются как есть. Никаких внешних ассетов из сети.
11. Игра проходится целиком до любой из концовок без ошибок в консоли.
12. В титрах/подвале указана команда SPDubstep.
